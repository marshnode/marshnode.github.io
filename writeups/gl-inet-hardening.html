GL-iNet Flint 2 Router Hardening & Troubleshooting

GL-MT6000 OpenWrt Firewall DNS over HTTPS Security Hardening ✓ Completed

Security hardening of a GL-iNet Flint 2 router running OpenWrt firmware — restricting admin access, encrypting DNS traffic, and implementing custom firewall rules to reduce the attack surface. Also documents a real-world troubleshooting case study where a misconfigured firewall rule caused full DHCP failure across the network, requiring systematic diagnosis and recovery.

Component Details
DeviceGL-iNet Flint 2 (GL-MT6000)
FirmwareOpenWrt (latest stable)
Admin AccessHTTPS via 192.168.8.1
DNS ProviderCloudflare (1.1.1.1 / 1.0.0.1)
NetworkHome LAN with multiple connected devices

// objectives

What This Lab Covers

  • Restrict router admin panel to HTTPS only
  • Encrypt all DNS queries using DNS over HTTPS (DoH)
  • Enable DNS rebinding protection to prevent DNS-based attacks
  • Implement custom firewall rules to control inbound and outbound traffic
  • Document and recover from a self-caused network misconfiguration

// hardening · step 01

Enforcing HTTPS-Only Admin Access

By default, the GL-iNet admin panel is accessible over both HTTP and HTTPS. Allowing unencrypted HTTP access means admin credentials could be intercepted on the local network.

What I Did

  • Logged into the admin panel at 192.168.8.1
  • Navigated to System → Administration
  • Disabled HTTP access and enforced HTTPS only
  • Verified that navigating to http://192.168.8.1 no longer loaded the panel and redirected to HTTPS

Why It Matters

Even on a home network, enforcing HTTPS ensures admin credentials are never transmitted in plaintext. This is especially important if the network has multiple users or guest devices.

// hardening · step 02

Configuring Cloudflare DNS over HTTPS

Standard DNS queries are sent in plaintext over UDP port 53, meaning your ISP or anyone on the network can see every domain you look up. DNS over HTTPS encrypts these queries inside HTTPS traffic.

What I Did

  • Navigated to Network → DNS in the OpenWrt admin panel
  • Disabled the default ISP-assigned DNS servers
  • Configured Cloudflare's DoH endpoints: Primary https://1.1.1.1/dns-query · Secondary https://1.0.0.1/dns-query
  • Enabled the DoH client within OpenWrt
  • Verified DNS resolution was functioning correctly after changes

Why It Matters

Encrypting DNS traffic prevents ISP-level DNS snooping and protects against DNS hijacking attacks where malicious actors redirect traffic by intercepting DNS queries.

// hardening · step 03

Enabling DNS Rebinding Protection

DNS rebinding is an attack technique where a malicious website causes a victim's browser to make requests to internal network resources, such as a router admin panel, by manipulating DNS responses.

What I Did

  • Located the DNS rebinding protection setting under Network → DNS
  • Enabled rebinding protection in OpenWrt
  • Configured the router to reject DNS responses that resolve public domain names to private IP ranges (RFC 1918: 10.x.x.x, 172.16.x.x, 192.168.x.x)

Why It Matters

Without this protection, an attacker could craft a malicious website that tricks the browser into attacking devices on the local network, including the router itself.

// hardening · step 04

Custom Firewall Rules

After completing the above hardening steps, additional firewall rules were implemented through the OpenWrt firewall configuration to further restrict unnecessary traffic and tighten the network perimeter. Rules were applied via SSH into the router using UCI commands directly in the OpenWrt shell.

Rules Implemented

  • Blocked unsolicited inbound traffic from WAN to LAN
  • Restricted admin panel access to specific trusted devices on the LAN
  • Configured logging for dropped packets to assist with future monitoring
  • Set default policies: WAN input = DROP · WAN forward = DROP · LAN = ACCEPT

// incident report

Self-Caused DHCP Failure

⚠ Incident Summary

While implementing a custom firewall rule to restrict inbound traffic, a rule was accidentally introduced that blocked DHCP traffic on the LAN interface. Within minutes, connected devices began losing their IP addresses and were unable to obtain new leases. The network went down completely.

Impact

  • All devices on the LAN lost internet connectivity
  • New devices were unable to connect to the network
  • Router admin panel was briefly inaccessible due to IP loss on the client machine

Diagnosis Process

STEP 01

Identify the Symptom

Devices showed "No IP address" or self-assigned 169.254.x.x APIPA addresses, which immediately indicated DHCP failure rather than a routing or WAN issue.

STEP 02

Isolate the Cause

Connected a device directly via ethernet and attempted a manual static IP assignment, which succeeded. Confirmed the router was functional but DHCP leasing was broken.

STEP 03

Access the Router

Assigned a static IP manually in the 192.168.8.x range to regain access to the admin panel at 192.168.8.1 without needing a DHCP lease.

STEP 04

Review Firewall Rules

Reviewed recently applied rules inside OpenWrt firewall config. Identified a rule incorrectly blocking UDP port 67 and 68, the ports used by DHCP for server and client communication.

STEP 05

Fix and Verify

Removed the offending rule, restarted the DHCP service, and verified that devices began obtaining IP addresses correctly. Full connectivity restored within minutes.

Root Cause

The firewall rule was written too broadly. The intent was to block unsolicited inbound traffic from external sources, but the rule inadvertently applied to internal LAN traffic as well, blocking the DHCP broadcast packets that devices use to request IP addresses.

Fix Applied

Rewrote the rule with a narrower scope, explicitly specifying the WAN interface as the target rather than applying it globally. Added an explicit ACCEPT rule for DHCP traffic on the LAN interface to prevent similar issues in the future.

// key takeaways

Lessons Learned

  • Always scope firewall rules precisely. A rule that is too broad can have unintended consequences across multiple services and interfaces.
  • Know your recovery path before applying changes. Having a plan to access the router via static IP saved significant time during recovery.
  • DHCP runs on UDP 67/68. Blocking these ports on the wrong interface will silently kill all dynamic IP assignment on the network.
  • Test in stages. Applying multiple firewall rules at once makes it harder to identify which rule caused a problem. Apply and verify one at a time.
  • Document as you go. The troubleshooting process was faster because I knew exactly what changes had been made and in what order.
