TechCorp HQ Enterprise Network Design

Cisco Packet Tracer VLANs OSPF EtherChannel NAT/PAT Cisco IOS ✓ Completed

Full design, build, and verification of an enterprise network for a fictional company — TechCorp HQ. Starting from a blank Packet Tracer canvas, the topology was designed, every device cabled, every protocol configured, and end-to-end connectivity verified across all three departments. A CCNA-level lab mirroring real-world infrastructure, with every configuration decision documented alongside the reasoning behind it — not just the commands.

// topology

Network Topology

The network follows a three-tier hierarchical model:

  • Edge Router (R1): Cisco 2911. Handles NAT/PAT and OSPF routing toward the internet.
  • Core Switch (SW-Core): Cisco 3560 multilayer switch. Handles inter-VLAN routing and OSPF.
  • Access Switches (SW-Access1/2/3): Cisco 2960 Layer 2 switches. Connect end devices to their VLANs.

Each access switch connects to the core via two physical cables bundled into a single logical EtherChannel link using LACP for redundancy and increased bandwidth.

// stage 01

VLAN Creation

Created four VLANs on every switch: Management (10), Engineering (20), Finance (30), and Native (99). Applied on SW-Core and all three access switches.

Why It Matters

Without VLANs, every device shares the same broadcast domain — every ARP request and DHCP broadcast goes to every device, degrading performance and creating security risks. Finance traffic would be visible to Engineering. VLANs contain traffic within each department at Layer 2.

VLAN 99 was designated as the native VLAN. Best practice is to move the native VLAN away from the default VLAN 1, which is a known vector for VLAN hopping attacks.

// stage 02

Trunk Ports (802.1Q)

Configured uplink ports between SW-Core and each access switch as 802.1Q trunk ports on both ends. Every trunk allows VLANs 10, 20, 30, and 99 with native VLAN 99.

Why It Matters

A standard access port carries only one VLAN. Without trunking, only one VLAN's traffic could cross the uplinks, defeating the purpose of VLAN segmentation. 802.1Q tagging inserts a 4-byte VLAN tag into each Ethernet frame so all three department VLANs can share a single physical cable.

Troubleshooting Encountered

Native VLAN Mismatch — %CDP-4-NATIVE_VLAN_MISMATCH

Access switches still had default native VLAN 1 while SW-Core was configured for VLAN 99. Fix: apply native VLAN 99 on all access switch uplink ports. Trunk configuration is always a two-sided commitment.

// stage 03

Access Port Assignment

Assigned the PC-facing port on each access switch to its correct department VLAN: PC0 → VLAN 10, PC1 → VLAN 20, PC2 → VLAN 30. Each access port configured as non-trunk, belonging to exactly one VLAN.

Why It Matters

Without this step, all PCs would sit on VLAN 1 by default with no connectivity to the correct department subnets. Access port assignment is what actually places each end device into its logical network segment.

// stage 04

EtherChannel (LACP)

Bundled the two physical cables between SW-Core and each access switch into logical port channels using LACP active mode on both sides. Three port channels created: Po1 (SW-Core ↔ SW-Access1), Po2 (SW-Core ↔ SW-Access2), Po3 (SW-Core ↔ SW-Access3). Each configured as a trunk.

Why It Matters

Without EtherChannel, STP would detect a loop and block one of the two links — leaving one cable completely idle. EtherChannel makes STP see the bundle as a single logical link, both cables actively carry traffic (doubled bandwidth), and if one fails the other continues with no reconvergence. LACP was chosen as the open IEEE 802.3ad standard over Cisco's proprietary PAgP.

Troubleshooting Encountered

CANNOT_BUNDLE2 — VLAN Mask Mismatch

Ports were added to channel groups before trunk config was applied to physical interfaces. Fix: configure trunk encapsulation and allowed VLANs on physical interfaces first, then assign to channel group. The port channel inherits from member ports, not the other way around.

Suspended Ports — LACP Not Negotiating

Configuration was applied to only one side at a time. LACP times out waiting for the remote end. Fix: configure both sides in quick succession.

Administratively Disabled Interfaces

Fa0/1 and Fa0/4 on SW-Core were in shutdown state. Running no shutdown on both resolved Port-channel 1 remaining down despite full configuration.

// stage 05

Inter-VLAN Routing (SVIs)

Enabled IP routing on SW-Core and created a Switched Virtual Interface per VLAN, each assigned the gateway IP for its department. Static IPs assigned to each PC with their SVI as default gateway.

Why It Matters

VLANs are isolated by design — a PC in VLAN 10 cannot natively reach VLAN 20. SVIs are virtual Layer 3 interfaces inside the multilayer switch. Traffic destined for another VLAN hits the SVI, SW-Core looks up the destination in its routing table and forwards at hardware speed. This Layer 3 switching approach is far more efficient than router-on-a-stick.

// stage 06

OSPF Dynamic Routing

Configured a routed point-to-point link between SW-Core and R1 using a /30 subnet (10.0.0.0/30). SW-Core Fa0/7 converted from switchport to routed interface. Both devices configured with OSPF process 1 in Area 0. R1 configured to advertise a default route via default-information originate.

Why It Matters

Without OSPF, SW-Core and R1 only know their directly connected networks. Static routes don't scale. OSPF neighbor adjacency lets R1 automatically learn all three VLAN subnets from SW-Core, and SW-Core learns the default route from R1 — directing all unknown traffic toward the internet. A /30 subnet provides exactly two usable addresses for the point-to-point link with no waste.

Troubleshooting Encountered

Default Route Not Propagating — Gateway of Last Resort Not Set

default-information originate only advertises a default route if R1 itself has one. Fix: add a static default route on R1 pointing to the ISP gateway first, then OSPF advertises it to SW-Core.

// stage 07

NAT / PAT

Configured Port Address Translation on R1 to allow all internal hosts across all three VLANs to reach the internet using R1's single public IP (203.0.113.1). A standard ACL defined the internal networks eligible for translation. NAT overload applied to the WAN interface.

Why It Matters

The 192.168.x.x VLAN subnets are RFC 1918 private addresses — not routable on the public internet. NAT translates private source addresses to the router's public IP as packets leave the network. PAT extends this by using port numbers to differentiate thousands of simultaneous connections from different internal hosts — all sharing one public IP. This is exactly how real-world home routers and corporate gateways operate.
